## 做了什么

- 为 `src/edane_full_pipeline.py` 增加统一的进度输出 `_progress()`（带时间戳、flush），在长任务阶段持续给终端反馈。
- 在关键阶段增加进度提示：
  - file 模式解析路径、构建图输入、模型构建、初始 fit、每个 snapshot 的更新与评估。
  - CSV 读取阶段增加“读取完成/大文件周期性行数”提示，避免用户误以为卡死。
- 为避免 Windows 终端编码导致中文乱码，新增的进度提示优先使用 ASCII/英文描述。

## 变更前快照（问题表现）

- 运行 `python src/edane_full_pipeline.py --mode file --model dtformer --snapshots 3 --max-nodes 15000` 等命令时，较长时间终端无任何输出，用户无法判断是否在运行、卡住或还剩多久。

## 变更内容

- `src/edane_full_pipeline.py`
  - 新增 `_progress(message: str)`。
  - 改造 `read_csv_dict()`：在读取结束时输出行数与耗时；对较大文件按阈值周期性输出已读取行数。
  - 在 `build_graph_from_files()` 与 `run_pipeline()` 增加阶段性进度输出；每个快照输出 batch_events、更新耗时、开始评估等。
  - 在 `evaluate_snapshot()` / `softmax_logreg_predict()` 增加评估期进度输出：
    - 输出 labeled_nodes / classes 等评估规模信息；
    - repeated_stratified 时按 repeat 输出进度；
    - logreg 训练按 epoch 分位输出进度（仅在较大 N 或类别多时触发），避免长时间无反馈。

## 影响范围与风险

- 仅增加日志输出，不改模型算法与指标计算逻辑。
- 风险：输出变多（但控制为“关键阶段 + 大文件周期性输出”，避免刷屏）。

## 验证

- `python -m py_compile src/edane_full_pipeline.py` 通过。
- 通过小规模 file 模式运行进行人工验证：终端能持续看到 “Loading CSV / fit / snapshot update / evaluation” 等进度信息；且在 logreg 训练期间能看到 epoch 进度（大规模时）。
